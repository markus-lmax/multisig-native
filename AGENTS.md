# AGENTS.md

## Overview

Native Solana multisig program (no Anchor framework). Rust on-chain program in `program/`, TypeScript client SDK in `ts/`, integration tests in `tests/`. A re-implementation of [LMAX Multisig](https://github.com/LMAX-Exchange/multisig) without Anchor.

## Architecture

- **On-chain program** (`program/src/`): Borsh-serialized instructions dispatched via a single `process_instruction` entrypoint in `processor.rs`. Each instruction has its own file under `instructions/` following the pattern: parse accounts with `next_account_info`, validate, execute. Shared validation helpers live in `instructions/common.rs`.
- **State accounts** (`program/src/state/`): `Multisig` (owners, threshold, nonce, padding for re-expansion) and `Transaction` (instructions, per-owner signers bitmask, owner_set_seqno). The `padding` field in `Multisig` preserves account size when shrinking owners, allowing later re-expansion without reallocation.
- **PDA signer**: The multisig PDA is derived from `[multisig_account_key, &[nonce]]`. The PDA acts as the authority for governed resources (mint authority, token owner, etc.) and signs via `invoke_signed` during execution.
- **Instruction enum ordering matters**: `MultisigInstruction` variants are Borsh-discriminated by index (0=CreateMultisig, 1=SetOwners, ... 7=CancelTransaction). The TS enum in `ts/instructions/index.ts` must stay in sync.
- **TS client SDK** (`ts/`): Each instruction builder in `ts/instructions/` manually constructs Borsh-serialized data using schemas with an `instructionDiscriminator: "u8"` prefix. State deserialization uses `borsh` v2 schemas in `ts/state/`. The `Assignable` base class (`ts/assignable.ts`) is a generic property-bag used by Borsh.
- **Test DSL** (`ts/multisigDsl.ts`): `MultisigDsl` encapsulates the full propose→approve→execute lifecycle. Tests use `solana-bankrun` (`ProgramTestContext`) — no live validator needed. The program binary is loaded from `program/target/so/multisig_native.so`.

## Build & Test

```bash
pnpm install          # once
pnpm build            # cargo build-sbf + shank idl generation
pnpm test             # ts-mocha against bankrun (requires prior build)
pnpm build-and-test   # both in sequence
```

- `SBF_OUT_DIR=program/target/so` is set automatically by the test script so bankrun finds the `.so` file.
- IDL is generated by `shank` into `program/target/idl/multisig_native.json`.
- Tests use `node:test` (`describe`/`test`) with `chai` assertions, run via `ts-mocha`. Config: `tests/tsconfig.test.json`.

## Conventions

- **Error handling**: Use `assert_that`, `assert_present`, `assert_success` from `errors.rs` — not raw `if`/`return Err`. Errors are `MultisigError` enum variants mapped to `ProgramError::Custom(variant as u32)`. Tests assert on hex error codes like `custom program error: 0x0`.
- **Instruction pattern** (Rust): define a `XxxInstruction` struct with `BorshSerialize`/`BorshDeserialize`, a public `fn xxx(program_id, accounts, instruction)` handler, and a private `fn validate(...)`.
- **Instruction pattern** (TS): export a `createXxxInstruction` function that builds a `TransactionInstruction` with Borsh-serialized data. Schema struct must include `instructionDiscriminator` as first field.
- **Account ordering**: Accounts are parsed positionally via `next_account_info`. The order is documented in `#[account(...)]` Shank attributes on `MultisigInstruction` variants in `processor.rs` — this is the source of truth.
- **Execute requires remaining accounts**: `ExecuteTransaction` appends all accounts referenced by the inner instructions as remaining accounts after the 5 fixed accounts. The TS side deduplicates and marks the multisig signer as `isSigner: false` (PDA signs via `invoke_signed`).
- **owner_set_seqno**: Incremented on owner changes; transactions store it at proposal time. Stale transactions (mismatched seqno) are rejected on approve/execute.
